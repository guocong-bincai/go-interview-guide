# 信封估算法（Back-of-Envelope）

> 考察频率：★★★☆☆  难度：★★★★☆
> 核心能力：快速估算系统容量，不需要精确计算，用估算证明设计可行

## 核心公式

```
QPS = 用户数 × 日活(DAU) × 人均请求数 / 86400秒
存储 = 用户数 × 人均数据量 × 副本系数
带宽 = QPS × 单次请求大小
```

---

## 快速估算四步法

### Step 1：确定用户规模

| 规模 | 用户量 | 说明 |
|------|--------|------|
| 小型 | < 100 万 | 创业公司 |
| 中型 | 100 万 - 1 亿 | 成长型产品 |
| 大型 | 1 亿 - 10 亿 | 头部产品 |
| 超大 | > 10 亿 | BAT 级别 |

### Step 2：估算 DAU（日活）

```
DAU = 用户数 × 活跃率

典型活跃率：
- 社交/短视频：30-50%
- 电商：10-20%
- B2B 工具：5-10%
- 企业内部：50-80%
```

### Step 3：估算 QPS

```
峰值 QPS = 平均 QPS × 峰值系数

峰值系数：
- 普通系统：2-3x
- 秒杀/活动：10-100x
- 春晚/双11：1000x+
```

### Step 4：容量计算

```
存储 = 用户数 × 人均数据量
带宽 = QPS × 请求大小
机器数 = QPS / 单机 QPS
```

## 实战估算

### 案例一：设计一个微博 Feed 系统

**已知条件**：
- 用户数：1 亿
- DAU：3000 万（30% 活跃率）
- 用户平均关注：200 人
- Feed 刷新频率：每用户每天 20 次
- 单条 Feed 大小：1KB

**Step 1：计算 QPS**

```
日总请求 = 3000万 × 20 = 6亿次
平均 QPS = 6亿 / 86400 ≈ 7000 QPS
峰值 QPS = 7000 × 5 = 35000 QPS（峰值系数按5x）
```

**Step 2：存储估算**

```
用户 Feed 存储：
- 每人存储最近 1000 条 Feed
- 总存储 = 1亿 × 1000 × 1KB = 100TB
- 副本系数 3 = 300TB

用户关系存储：
- 1亿 × 200 × 元数据 ≈ 10GB
```

**Step 3：带宽估算**

```
Feed 请求带宽 = 35000 × 200 × 1KB ≈ 7GB/s ≈ 56Gbps
（每个请求返回200条Feed，每条1KB）
```

**Step 4：机器估算**

```
QPS 需求：峰值 35000 QPS
单机处理能力（Redis 缓存）：10万 QPS
需要机器：35000 / 100000 ≈ 1台（纯缓存）
考虑 CPU 开销和冗余：5-10台
```

**结论**：
```
存储：300TB（3副本）
峰值 QPS：35000
机器：5-10 台应用服务器 + Redis 集群
```

### 案例二：设计一个短链系统

**已知条件**：
- 日均跳转：1 亿次
- 短链平均长度：7 字符
- 单次跳转耗时：< 50ms

**Step 1：QPS 计算**

```
平均 QPS = 1亿 / 86400 ≈ 1157 QPS
峰值 QPS = 1157 × 10 = 11570 QPS（按10x峰值）
```

**Step 2：存储估算**

```
1亿条短链记录：
每条记录：短码(7B) + 原URL(100B) + 额外(20B) ≈ 130B
总存储 = 1亿 × 130B ≈ 13GB
考虑索引和副本 = 100GB 级别
```

**Step 3：Redis 缓存**

```
热点数据：访问量前 20% 的短链占总流量 80%
缓存 20% = 缓存 2000万条
Redis 内存 = 2000万 × 130B ≈ 2.6GB
```

**Step 4：机器估算**

```
单机 QPS：5万（纯读 Redis）
需要机器：11570 / 50000 ≈ 1台
考虑高可用：2-3台
```

**结论**：
```
存储：100GB（HBase/Redis）
缓存：3GB Redis
机器：2-3 台跳转服务
```

### 案例三：设计秒杀系统

**已知条件**：
- 商品 1000 件
- 参与用户：100 万
- 持续时间：10 分钟

**Step 1：库存校验 QPS**

```
峰值 QPS = 100万 / (10×60) ≈ 1667 QPS
理论值看起来不大，但库存扣减集中在前 10 秒
实际峰值 QPS = 100万 / 10 = 10万 QPS
```

**Step 2：Redis 容量**

```
QPS 需求：10万 QPS
单机 Redis：10万 QPS 刚好
→ 至少需要 2 台 Redis（主从）+ 1 台 Proxy
```

**Step 3：带宽估算**

```
请求大小：userID + productID ≈ 100B
响应大小：库存结果 ≈ 50B
总带宽 = 10万 × (100B + 50B) / 秒 ≈ 15MB/s ≈ 120Mbps
```

**Step 4：数据库估算**

```
写订单量 = 1000 件
10 分钟内完成 = 1000 / 600 ≈ 2 TPS（几乎无压力）
MySQL 完全可处理
```

**结论**：
```
Redis 集群：2 台（主从），处理 10万 QPS
消息队列：Kafka，分散瞬时流量
数据库：MySQL，无压力
机器：5-10 台 Web 服务
```

## 常见估算参数速查

### 单机处理能力

| 类型 | QPS | 说明 |
|------|-----|------|
| 简单读（内存） | 10-20 万 | 直接读 Redis |
| 简单写（内存） | 5-10 万 | 直接写 Redis |
| 数据库简单查询 | 1-5 千 | 有索引的主键查询 |
| 数据库复杂查询 | 100-500 | 多表 Join、聚合 |
| 磁盘顺序写 | 500 MB/s | 顺序写入日志 |

### 存储估算

| 数据类型 | 大小 |
|---------|------|
| 整型数字 | 8B |
| UUID | 16B |
| 用户头像（压缩后） | 50KB |
| 短视频 | 1-10MB |
| 一条微博 | 1-5KB |
| 关系图谱节点 | 100-500B |

### 带宽估算

| 带宽 | 速度 |
|------|------|
| 千兆网卡 | 125 MB/s ≈ 1 Gbps |
| 万兆网卡 | 1.25 GB/s ≈ 10 Gbps |
| 内网 VPC | 通常够用 |
| 公网出口 | 按需付费，注意成本 |

### 可用存储

| 存储类型 | 单机容量 | 特点 |
|---------|---------|------|
| SSD | 1-10 TB | 快，贵 |
| HDD | 10-40 TB | 便宜，慢 |
| Redis | 64-256 GB | 极快，内存有限 |
| HBase | TB-PB 级别 | 分布式，适合历史数据 |

## 面试话术

**Q：如何快速估算双十一的系统容量？**

> 先估算交易量：假设 10 亿人参与，1 亿人下单，平均客单价 200 元，峰值 QPS = 1 亿 / 3600 ≈ 3 万（集中在 2 小时内）。实际峰值可能是 10 倍 = 30 万 QPS。数据库 TPS = 实际下单量 / 时间 = 1 亿 / 7200 ≈ 1.4 万 TPS。MySQL 单机可以处理 5000-1 万 TPS，所以至少需要 2-3 台 DB 加上读写分离。缓存需要处理读 QPS，30 万 QPS × 读多写少（10:1），Redis 单机 10 万 QPS，需要 5-6 台 Redis 集群。

**Q：数据量很大怎么办？**

> 分而治之：1）冷热分离，热数据放 Redis/MySQL，冷数据放对象存储/大数据；2）分库分表，按用户 ID 或时间分片；3）数据分层，多级缓存（本地缓存 → Redis → DB）；4）归档历史数据，减少单库数据量。

**Q：带宽不够怎么办？**

> 1）CDN 加速静态资源；2）数据压缩（gzip、protobuf）；3）合并请求，减少 RTT；4）升级带宽（贵）；5）异地多活，本地访问就近节点。
