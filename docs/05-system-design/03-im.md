## IM 系统核心指标

| 指标 | 说明 | 目标 |
|------|------|------|
| **送达率** | 消息成功到达对方的比例 | > 99.99% |
| **延迟** | 消息发出到对方看到的时延 | < 100ms |
| **并发连接** | 同时在线用户数 | 百万级 |
| **消息顺序** | 同一会话内消息顺序一致 | 必须保证 |

## 消息模型

### 单聊 vs 群聊

**单聊：** 一对一关系，消息投递路径清晰
```
A → 消息网关 → 消息存储 → 消息推送 → B
```

**群聊：** 一对多，需要消息扩散写
```
A → 消息网关 → 消息存储 → 为每个群成员生成消息索引 → 推送通知
```

### 消息存储模型

```go

    MsgID      string    `gorm:"primaryKey"`  // 全局唯一 ID
```

## 消息可靠投递

### 三消息投递协议

**1. 写扩散（Write Fan-out）：**
- 写消息到每个接收者的消息列表
- 优点：读快，缺点：写慢（群聊消息量大）
- 适合：群成员少（< 100人）

**2. 读扩散（Read Fan-out）：**
- 写消息到一个中央消息表
- 读时拉取所有相关消息并合并
- 优点：写快，缺点：读慢
- 适合：群多、单聊多

**3. 混合模式（推拉结合）：**
```
小群（< 20人）：写扩散
大群（> 20人）：读扩散 + 拉取最新消息
```

### 消息 ID 设计（雪花算法 + 分片）

```go
// 消息 ID = 时间戳 + 用户分片 + 递增号
func GenerateMsgID(userID string) string {
    timestamp := time.Now().UnixMilli()
    shardID := hash(userID) % 1024
    seq := atomic.AddInt64(&seqCounter, 1) & 0xFFF
    return fmt.Sprintf("%d-%d-%d", timestamp, shardID, seq)
}
```

### 消息拉取策略

**1. 启动时全量拉取：**
```
App 启动 → 拉取最近 N 条离线消息 → 显示
```

**2. 增量同步：**
```
客户端记录 lastSeq → 启动时用 lastSeq + 1 增量拉取
```

**3. 拉取合并：**
```go
// 客户端按会话批量拉取，避免频繁请求
func PullOfflineMessages(userID string, lastSeq int64) ([]*Message, error) {
    // 一次拉取多个会话的最新消息
    rows, err := db.Raw(`
        SELECT m.* FROM messages m
        JOIN (
            SELECT conversation_id, MAX(seq_id) as max_seq
            FROM conversation_messages
            WHERE user_id = ? AND seq_id > ?
            GROUP BY conversation_id
        ) t ON m.conversation_id = t.conversation_id
        WHERE m.seq_id <= t.max_seq
        ORDER BY m.seq_id DESC
        LIMIT 100
    `, userID, lastSeq).Rows()
}
```

```go
type ConversationUnread struct {
    ConversationID string
    UserID         string

    LastReadSeq    int64

// 实时更新未读数
func (s *IMService) OnMessageDelivered(msg *Message) error {
    // 遍历接收者列表
    for _, userID := range msg.Receivers {
        // 在线用户：WebSocket 推送
        if s.isOnline(userID) {
            s.wsServer.SendToUser(userID, msg)
        } else {
            // 离线用户：写入离线消息表
            s.offlineRepo.Add(ctx, userID, msg)
            // 发送离线推送
            s.pushService.Push(userID, msg)
        }
        
        // 更新未读计数
        s.redis.HincrBy(
            fmt.Sprintf("unread:%s:%s", userID, msg.ConversationID),
            "count", 1)
    }
    return nil
}
```

## 消息顺序保证

### 单聊：SeqID 保证顺序

```go
// 每条消息分配严格递增的 SeqID
func GenerateSeqID(conversationID string) int64 {
    return redis.Incr("seq:" + conversationID)
}

// 接收端按 SeqID 排序显示
func GetMessages(conversationID string, limit int) ([]*Message, error) {
    rows, _ := db.Query(`
        SELECT * FROM messages 
        WHERE conversation_id = ?
        ORDER BY seq_id DESC
        LIMIT ?
    `, conversationID, limit)
    
    msgs := sortBySeqID(rows)  // 保证顺序
    return msgs, nil
}
```

## Go WebSocket 实现

### 长连接网关

```go
type Gateway struct {
    // 用户 ID → WebSocket 连接
    connections sync.Map
    
    // 连接的心跳
    heartbeatInterval time.Duration
}

func (g *Gateway) HandleWS(ctx context.Context, userID string, conn *websocket.Conn) {
    // 注册连接
    g.connections.Store(userID, conn)
    defer g.connections.Delete(userID, conn)
    
    // 启动心跳
    go g.heartbeat(ctx, userID, conn)
    
    // 读消息
    for {
        _, msgBytes, err := conn.ReadMessage()
        if err != nil {
            break
        }
        g.handleMessage(ctx, userID, msgBytes)
    }
}

func (g *Gateway) handleMessage(ctx context.Context, userID string, msgBytes []byte) {
    var msg WSMessage
    json.Unmarshal(msgBytes, &msg)
    
    switch msg.Type {
    case "chat_msg":
        // 消息持久化 + 扩散
        s.chatService.SendMessage(ctx, &msg)
    case "read_receipt":
        s.chatService.MarkRead(ctx, userID, msg.ConvID, msg.ReadSeq)
    }
}
```

## 常见问题

### Q：消息重复消费如何处理？

> **消费端幂等**：用 Redis SETNX 记录 MsgID + 处理时间戳；**DB 层唯一约束**：MsgID 设唯一索引，重复插入报错。

### Q：消息延迟怎么排查？

> 1. 看推送链路延迟（网关 → MQ → 消费者）；2. 看离线消息堆积量；3. 看 APNs 推送延迟（苹果自带延迟）；4. 排查网络问题。

### Q：群聊消息量大的优化？

> **大群不写扩散**：只写中央消息表，成员拉取时合并；**历史消息分页**：按时间分页，拉取最近 N 条；**热点优化**：热门群消息加缓存。

### Q：IM 系统用什么协议？

> **WebSocket**（主流）+ **MQTT**（省电，适合移动端）；**自定义协议**优于 HTTP（避免频繁建连）；传输层用 **TLS**。
