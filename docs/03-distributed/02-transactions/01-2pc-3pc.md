# 2PC / 3PC 分布式事务协议

> 考察频率：★★★★☆  难度：★★★★☆

## 2PC（Two-Phase Commit）

### 核心角色

| 角色 | 职责 |
|------|------|
| **协调者（Coordinator）** | 发起提交决策，管理整个事务 |
| **参与者（Participant）** | 执行本地事务，接受协调者指令 |

### 两阶段流程

```
 Phase 1：请求提交（Prepare）
   协调者 → 所有参与者：Prepare 请求
   参与者 → 协调者：Vote（YES/NO）

 Phase 2：提交/回滚（Commit/Rollback）
   如果全部 YES：
     协调者 → 所有参与者：Commit
     参与者 → 协调者：ACK
   如果有 NO：
     协调者 → 所有参与者：Rollback
     参与者 → 协调者：ACK
```

### 图示

```
协调者                    参与者A                   参与者B
   │                         │                         │
   │──── Prepare ──────────→ │                         │
   │                         │                         │
   │──── Prepare ───────────────────────────────────→ │
   │                         │                         │
   │←──── YES ─────────────── │                         │
   │←──── YES ─────────────────────────────────────── │
   │                         │                         │
   │──── Commit ───────────→ │                         │
   │──── Commit ────────────────────────────────────→ │
   │                         │                         │
   │←──── ACK ────────────── │                         │
   │←──── ACK ───────────────────────────────────── │
```

### Go 实现

```go
type Coordinator struct {
    participants []*Participant
    txn          *sql.Tx
}

type Participant struct {
    id     string
    status TransactionStatus
}

type TransactionStatus int

const (
    StatusPrepared TransactionStatus = iota
    StatusCommitted
    StatusAborted
)

func (c *Coordinator) Execute() error {
    // Phase 1: Prepare
    for _, p := range c.participants {
        if err := p.Prepare(); err != nil {
            return c.Abort() // 有失败，全部回滚
        }
    }

    // Phase 2: Commit
    return c.Commit()
}

func (c *Coordinator) Commit() error {
    for _, p := range c.participants {
        if err := p.Commit(); err != nil {
            // 记录日志，人工干预（无法自动回滚其他参与者）
            log.Printf("participant %s commit failed: %v", p.id, err)
        }
    }
    return nil
}

func (p *Participant) Prepare() error {
    // 本地事务预提交（不释放锁）
    tx, err := db.Begin()
    if err != nil {
        return err
    }
    // 执行 SQL，但不 commit/rollback
    _, err = tx.Exec("UPDATE account SET balance = balance - ? WHERE id = ?", p.amount, p.accountID)
    if err != nil {
        tx.Rollback()
        return err
    }
    // 预提交成功，持有锁，等待协调者指令
    p.tx = tx
    p.status = StatusPrepared
    return nil
}

func (p *Participant) Commit() error {
    p.tx.Commit()
    p.status = StatusCommitted
    return nil
}
```

## 2PC 的问题

### 1. 同步阻塞

参与者 Prepare 后持有锁，等待协调者 Commit/Rollback。期间其他事务无法访问这些行。

**解决**：设置超时，协调者超时后自动 Rollback。

### 2. 协调者单点故障

协调者在 Phase 2 开始前崩溃：参与者等待 Commit，永远阻塞。

```
协调者（A）                         参与者（B）
    │                                  │
    │──── Prepare ──────────────────→ │
    │                                  │ ← Prepare 成功，持有锁
    │              [A 崩溃]            │
    │                                  │ ← 永久等待 B.DO...
```

### 3. 数据不一致

协调者 Commit 消息部分送达：
- B 收到 Commit，执行成功
- A 未收到 Commit（崩溃恢复后认为自己应该 Rollback）

## 3PC（Three-Phase Commit）

3PC 在 2PC 基础上增加 **CanCommit** 阶段，将提交分为三步：

```
Phase 1：CanCommit（询问）
  协调者 → 参与者：CanCommit？
  参与者 → 协调者：YES/NO

Phase 2：PreCommit（预提交）
  如果全部 YES：
    协调者 → 参与者：PreCommit
    参与者 → 协调者：ACK（已准备好提交）
  如果有 NO：
    协调者 → 参与者：Abort

Phase 3：DoCommit（真正提交）
  协调者 → 参与者：DoCommit
  参与者 → 协调者：ACK
```

### 与 2PC 的区别

| 对比 | 2PC | 3PC |
|------|-----|-----|
| 阶段数 | 2 | 3 |
| 同步阻塞 | 有（Prepare 阶段就持锁） | 无（PreCommit 才持锁） |
| 协调者单点 | 存在 | 部分解决 |
| 数据不一致 | 可能 | 概率更低 |
| 超时机制 | 无（需额外设计） | 有（各阶段都有超时） |

### 3PC 的改进点

```go
// 参与者超时处理
func (p *Participant) OnTimeout() {
    switch p.status {
    case StatusPrepared:
        // 3PC 特有：即使 Prepared，也可以超时 Rollback
        // 因为协调者还没真正提交
        p.Rollback()
    case StatusPreCommitted:
        // 不能自行决定，需要询问协调者
        // 或选择提交（最大努力）
        p.Commit()
    }
}
```

**3PC 不能完全解决数据不一致**：在 Phase 2 和 Phase 3 之间协调者崩溃，参与者超时后仍需自行决策。

## XA 协议

XA 是 2PC 的工业实现，用于**数据库层面**的分布式事务：

```go
import "github.com/go-sql-driver/mysql"

func xaTransaction() error {
    // 开启 XA 事务
    _, err := db.Exec("XA START 'txn1'")
    if err != nil {
        return err
    }

    _, err = db.Exec("UPDATE account SET balance = balance - 100 WHERE id = 1")
    if err != nil {
        db.Exec("XA END 'txn1'")
        db.Exec("XA ROLLBACK 'txn1'")
        return err
    }

    _, err = db.Exec("XA END 'txn1'")
    _, err = db.Exec("XA PREPARE 'txn1'")

    // 两阶段提交
    _, err = db.Exec("XA COMMIT 'txn1'")
    return err
}
```

### XA 的局限性

- 仅支持支持 XA 的数据库（MySQL InnoDB、PostgreSQL 等）
- 同步阻塞，性能差
- 协调者崩溃后恢复困难
- **不适合跨服务的分布式事务**（推荐 TCC / Saga）

## 面试话术

**Q：2PC 最大的问题是什么？**
> 同步阻塞 + 协调者单点。Prepare 阶段后参与者持有锁等待协调者决策，协调者崩溃会导致所有参与者永远等待。3PC 通过增加 CanCommit 阶段，把持锁时机推迟到 PreCommit 之后，减少了阻塞时间，但仍然无法完全避免数据不一致。

**Q：什么场景用 2PC/XA？**
> 强一致、对数据准确性要求极高的场景，如银行转账、证券交易。缺点是性能差、跨库长事务会锁住大量资源。在互联网高并发场景下，通常选择 TCC 或 Saga 代替。

**Q：2PC 和 TCC 的区别？**
> 2PC 是数据库层面的协议，靠数据库的 redo/undo 日志回滚；TCC 是业务层面的协议，靠预留（Try）、确认（Confirm）、取消（Cancel）三个接口回滚。TCC 不依赖数据库的分布式事务支持，更灵活，但需要业务代码实现三个接口。
