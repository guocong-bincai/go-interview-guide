# 技术规划

> 考察频率：★★☆☆☆  难度：★★★★☆  
> 高级工程师/架构师必备能力：技术规划能力

## 核心问题

技术规划考察什么：
1. **全局视野**：不只是写代码，要理解业务
2. **优先级判断**：什么先做、什么后做
3. **资源协调**：如何在有限资源下最大化价值

## 1. 技术 OKR 制定

### O（Objective）怎么写

```
O1：提升系统稳定性，SLA 达到 99.95%
O2：支撑业务增长，架构支持 10 倍流量
O3：提升团队效能，交付周期缩短 30%
```

**好的 O 的特征**：
- 有挑战但不空泛
- 和业务目标对齐
- 能用数字衡量

### KR（Key Results）怎么写

```markdown
# O1 的 KRs：
KR1：核心接口 P99 延迟 < 200ms（当前 500ms）
KR2：系统可用性从 99.9% 提升到 99.95%
KR3：故障平均恢复时间 MTTR < 30min（当前 2h）

# O2 的 KRs：
KR1：压测验证 10000 QPS 下系统稳定（当前 3000 QPS）
KR2：数据库读写分离上线，读 QPS 扩容 3 倍
KR3：缓存命中率 > 95%（当前 70%）
```

## 2. 年度/季度规划流程

### Step 1：现状盘点

```markdown
## 系统现状盘点

### 稳定性
- 近一年故障次数：x 次
- 平均故障时长：x 小时
- SLO 达成率：xx%

### 性能
- 当前峰值 QPS：xxx
- P99 延迟：xxx ms
- 资源利用率：CPU xx%，内存 xx%

### 技术债
- 测试覆盖率：xx%
- 文档完善度：xx%
- 核心模块依赖：...
```

### Step 2：目标对齐

```
业务目标（来自产品/管理层）
    ↓
技术支撑（技术团队承接）
    ↓
技术目标（OKR）
```

### Step 3：排期规划

```markdown
## Q2 技术规划

### P0（必须做）
- [ ] 数据库升级到 8.0，支持 JSON 索引
- [ ] 核心链路全链路追踪上线
- [ ] 订单服务分库分表

### P1（应该做）
- [ ] 缓存中间件统一（从 Redis Cluster 切换到 Codis）
- [ ] 消息队列从 RabbitMQ 迁移到 Kafka

### P2（可以做）
- [ ] 单元测试覆盖率从 40% 提升到 70%

```

## 3. 容量规划案例

### 如何估算需要多少资源

```go
// 信封估算法
func estimateResources() {
    // 1. 业务估算
    dailyActiveUsers := 1_000_000     // 日活
    avgRequestsPerUser := 50          // 人均请求
    peakFactor := 10                  // 峰值系数
    
    dailyRequests := dailyActiveUsers * avgRequestsPerUser
    peakQPS := float64(dailyRequests) / (24 * 3600) * peakFactor
    
    fmt.Printf("预估峰值 QPS: %.0f\n", peakQPS)
    // 输出：预估峰值 QPS: 579
    
    // 2. 资源估算
    // 假设：单台 4C8G 机器能扛 500 QPS
    machines := math.Ceil(peakQPS / 500)
    fmt.Printf("需要机器数: %.0f 台\n", machines)
    // 输出：需要机器数: 2 台（预留 1 台备用）
    
    // 3. 存储估算
    // 假设：每个用户每天产生 1KB 数据，保留 30 天
    storagePerDay := float64(dailyActiveUsers) * 1024
    storageTotal := storagePerDay * 30
    fmt.Printf("需要存储: %.2f GB\n", storageTotal / 1024 / 1024 / 1024)
    // 输出：需要存储: 27.96 GB
}
```

## 4. 技术选型决策

### 决策框架

```markdown
## 技术选型评估

### 选项：A 方案 vs B 方案

| 维度 | A 方案 | B 方案 |
|------|--------|--------|
| 开发成本 | 高（需要3人月） | 低（1人月） |
| 运维成本 | 高 | 低 |
| 扩展性 | 好 | 一般 |
| 社区活跃度 | 一般 | 活跃 |
| 学习成本 | 高 | 低 |

### 决策
选 B 方案。理由：
1. 业务窗口只有 1 个月，B 方案更快落地
2. B 方案社区活跃，出问题容易找到方案
3. 未来扩展性不足时再迁移

### 风险
- 未来迁移成本
- 应对措施：做好接口抽象，隔离厂商依赖
```

## 5. 面试话术

**Q：你们团队的技术规划是怎么做的？**

> 从业务目标出发，先盘点现状和问题，再制定目标。技术规划不是我想做什么就做什么，而是业务需要什么、我能解决什么。比如业务要推大促活动，技术规划就要提前做容量规划、限流保护、压测验证。

**Q：技术债怎么处理？**

> 技术债不能不管也不能全管。我的做法是：1）在每个迭代中留 20% 时间处理技术债；2）优先级按"影响范围 × 修复成本"评估，影响大且成本低的先修；3）新代码不许引入技术债，老代码借功能开发之机重构。

**Q：怎么协调业务和技术建设的优先级？**

> 用业务价值和技术成本做矩阵。紧急且重要的（线上故障）立即处理；重要不紧急的（技术债、架构优化）排进迭代；不重要的不做。我会定期和业务方对齐，让他们知道技术建设对业务的长期价值。

|------|---------|

| 排期规划 | 什么先做？资源够不够？ |
