# 架构演进：从单体到微服务的决策过程

> 面试频率：★★★★☆  考察角度：架构视野、演进思维、技术债务管理

---

## 1. 架构演进路线图

```
单体 → 垂直拆分 → 微服务 → 服务网格
  ↓        ↓          ↓         ↓
 小团队   中团队      大团队    超大团队
 1-10人   10-50人    50-200人  200+人
```

### 演进不是追新，是匹配业务阶段

| 业务阶段 | DAU | 团队规模 | 推荐架构 | 理由 |
|---------|-----|---------|---------|------|
| 冷启动 | < 1万 | 1-3 人 | 单体 | 快速试错，不需要微服务复杂度 |
| 成长期 | 1-50万 | 3-15人 | 巨石应用 + 缓存 + 数据库优化 | 稳定优先，延迟复杂度 |
| 增长期 | 50-500万 | 15-50人 | 业务拆分（按领域）+ MQ 解耦 | 团队规模到了，需要并行开发 |
| 成熟期 | > 500万 | 50-200人 | 微服务 + 容器化 + 弹性伸缩 | 需要独立扩缩容、独立部署 |
| 平台期 | > 5000万 | 200+人 | 服务网格 + 中台化 | 治理复杂度需要sidecar模式 |

## 2. 单体应用的困境与拆分信号

### 2.1 单体不是原罪

> "创业公司上来就搞微服务，是最常见的架构错误之一。" — Martin Fowler

单体的优势：
- **快**：不需要考虑服务间通信、本地函数调用
- **简单**：一个代码库，一个部署单元
- **低门槛**：新员工很快能上手

### 2.2 该拆分的信号

```
🚨 红灯信号（必须拆）：
- 团队 > 15 人，多个团队在同一个代码库冲突不断
- 单模块代码 > 50 万行，CI/CD 超过 30 分钟
- 某个业务模块需要独立扩缩容（如秒杀）
- 技术栈差异大（如 AI 模块用 Python，其他用 Go）

🟡 黄灯信号（考虑拆）：
- 多个团队在同一个表上做 Schema 变更，冲突频繁
- 核心业务和非核心业务部署在一起，非核心拖慢核心
- 频繁出现跨团队的技术栈争论
```

## 3. 垂直拆分：单体到微服务的过渡

### 3.1 按业务线拆分

```
原来：ecommerce（订单、用户、商品、支付全在一起）

拆分后：
├── user-service     （用户、认证、积分）
├── product-service  （商品、库存、分类）
├── order-service   （订单、结算）
├── payment-service （支付、退款）
└── notification-service（短信、推送、邮件）
```

### 3.2 拆分策略：绞杀者模式

不要推倒重来，用**绞杀者模式**逐步替换：

```
Step 1: 不改原有单体，在它前面加 API Gateway
Step 2: 新功能用微服务实现
Step 3: 逐渐把旧模块功能迁移到新服务
Step 4: 旧单体功能越来越少，最终下线
```

```go
// API Gateway 示例（以 Gin 为例）
func SetupRouter() *gin.Engine {
    r := gin.Default()

    // 旧单体服务（遗留）
    r.Any("/api/v1/*path", proxy.To("http://legacy-monolith"))

    // 新微服务
    userSvc :=.NewUserServiceClient()
    r.GET("/api/v2/users/:id", userSvc.GetUser)

    return r
}
```

## 4. 微服务拆分深度：领域驱动设计

### 4.1 DDD 限界上下文（Bounded Context）

不是按技术分层，而是按**业务领域**拆分：

```
❌ 按技术分层（错误）：
├── controller层
├── service层
├── dao层

✅ 按业务领域（正确）：
├── 库存领域（Inventory）
│   ├── 库存扣减
│   ├── 库存预占
│   └── 库存归还
├── 交易领域（Trading）
│   ├── 下单
│   ├── 取消
│   └── 超时关单
```

### 4.2 领域间如何划分边界

**高内聚、低耦合**原则：

| 判断标准 | 好（高内聚） | 差（低内聚） |
|---------|------------|-------------|
| 领域名词 | 同一业务语言 | 跨多个业务概念 |
| 变更频率 | 同一模块同时变更 | 一个字段被多个领域频繁修改 |
| 团队归属 | 同一团队负责 | 多个团队改同一模块 |

## 5. 架构演进真实案例

### 案例：电商平台 3 年演进史

**第一年（DAU 1 万）**
```
技术栈：Go 单体 + MySQL + Redis
架构：所有功能在一个进程中
问题：代码冲突多，部署相互影响
```

**第二年（DAU 50 万）**
```
技术栈：Go 单体 + Redis 缓存 + MySQL 主从
架构：引入缓存层，按业务垂直拆分服务
    user-service / product-service / order-service
治理：引入 Consul 做服务注册发现
```

**第三年（DAU 300 万）**
```
技术栈：微服务（Go）+ Kubernetes + Istio
架构：全量微服务化
    - 订单服务：独立扩缩容（P999 敏感）
    - 商品服务：读多写少，多副本
    - 支付服务：独立部署，数据强一致
治理：引入 Sentinel 做熔断降级
```

## 6. 架构演进的反模式

### 6.1 套娃式微服务

> "把一个服务拆成 5 个微服务，结果团队 3 个人，要维护 5 个服务、5 个数据库、5 套 CI、5 套监控。"

**教训**：微服务数量应该和团队规模成正比。

**经验公式**：
```
最优微服务数量 ≈ 团队人数 / 5~8
（每个微服务需要一个 SRE 轮值值班）
```

### 6.2 用微服务解决不存在的性能问题

```
"我们系统单机 QPS 500，你拆成微服务后调用链路变长，QPS 反而降到了 300。"
```

**教训**：性能问题先优化单机（缓存、数据库索引、代码逻辑），再考虑架构。

### 6.3 分布式计算陷阱

```
单机 1 秒能算完的东西，拆成 10 个微服务节点，
加上网络开销，可能变成 1.2 秒。
```

**教训**：分布式不是银弹，本地能解决的不需要分布式。

## 7. 面试高频追问

**Q：微服务拆分的原则是什么？**
> 按业务领域拆分，不是按技术层拆分；每个服务独立数据库；服务之间通过接口通信，不共享数据库。

**Q：什么时候该从单体迁移到微服务？**
> 团队规模 > 15 人、代码库 > 50 万行、部署频率要求 > 1次/天、独立扩缩容需求出现时。

**Q：服务拆分后数据一致性问题怎么解决？**
> 最终一致性优先：本地消息表 + 定时补偿；强一致场景用 Saga 模式或分布式事务（2PC/TCC）。

**Q：如何评估架构演进的效果？**
> 1. 部署频率是否提升（目标：每天可部署多次）；2. MTTR 是否缩短（目标：故障恢复 < 1小时）；3. 团队并行开发效率是否提升。
