# 项目复盘模板

> 考察频率：★★★☆☆  难度：★★★☆☆  
> 复盘能力是工程师成长的关键，也是面试中展示深度的好机会。

## 核心问题

项目复盘考察什么：
1. **结构化表达**：STAR + 数字量化
2. **反思深度**：不只是讲做了什么，要讲学到了什么
3. **可迁移性**：经验能不能复用

---

## 1. 复盘模板（STAR 扩展版）

### Situation - 背景

```
我负责的系统面临什么问题？
- 系统 QPS：xxx → 遇到瓶颈
- 团队规模：xx 人协作开发
- 时间窗口：必须在 xx 前完成
```

### Task - 目标

```
我的职责是什么？
- 独立负责 xxx 模块
- 配合 xxx 团队完成接口联调
- 需要在 2 周内交付
```

### Action - 行动

```
我具体做了什么？
1. 问题定位：用了什么工具、发现了什么根因
2. 方案设计：为什么选这个方案
3. 落地执行：关键里程碑
4. 跨团队协作：怎么协调资源
```

### Result - 结果

```
最终效果如何？
- 性能指标：延迟降低 xx%，吞吐提升 xx 倍
- 业务指标：转化率提升 xx%
- 工程指标：维护成本降低 xx%
```

### Learning - 反思（最重要！）

```
这次学到了什么？
- 做得好的：经验可以复用
- 做的不好的：下次怎么改进
- 可迁移：能不能做成通用方案
```

## 2. 真实案例：订单系统优化

### S - 背景

> 2024 年 Q2，我负责的订单系统在大促期间频繁超时，P99 延迟达到 8 秒，用户投诉率上升 30%。系统日均订单量 50 万，大促峰值 QPS 达到 3000+。

### T - 目标

> 在大促前（4 周）将 P99 延迟降到 500ms 以内，系统吞吐量提升到 10000 QPS。

### A - 行动

**1. 问题定位（1周）**
```bash
# 用 pprof 定位热点
go tool pprof -http=:8080 cpu.prof
# 发现：订单查询接口占了 60% CPU 时间
# 根因：N+1 查询 + 缺少索引

# 用链路追踪看慢请求
# 发现：订单服务 → 商品服务 → 用户服务 → 库存服务
#      每个下游超时 50-200ms 不等，串行调用
```

**2. 方案设计**
- **优化查询**：批量查询替代 N+1，增加数据库索引
- **并行化**：用 errgroup 并行调用下游服务
- **缓存**：热点数据加 Redis 缓存（商品信息、用户信息）
- **限流**：大促期间开启接口限流保护

**3. 落地执行**
- 第1周：问题定位 + 方案评审
- 第2周：缓存层 + 索引优化
- 第3周：并行化改造 + 压测
- 第4周：全链路压测 + 灰度发布

### R - 结果

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| P99 延迟 | 8000ms | 350ms | ↓95.6% |
| 系统 QPS | 3000 | 12000 | ↑4倍 |
| 数据库慢查询 | 2000+ /s | <10 /s | ↓99.5% |
| 大促零故障 | - | 完成 | - |

### L - 反思

**做得好：**
- 用数据驱动优化顺序，先解决最大的瓶颈（数据库）
- 压测验证了优化效果才上线

**可以改进：**
- 没有提前做容量规划，下次要提前 2 周开始压测
- 缓存一致性没有充分测试，上线后遇到一次缓存穿透

**可迁移：**
- 形成了"定位→优化→验证"的标准化排查流程
- 推广到其他服务：大促前统一做全链路压测

## 4. 面试话术

**Q：讲一个你做过的最有挑战的项目？**

> 用 STAR 框架，重点在 Action 和 Learning。很多候选人只讲做了什么，不讲为什么这么做、遇到了什么困难、怎么解决的。面试官更关心的是你的思考过程，而不是最终结果。

**Q：项目中遇到过最大的挫折是什么？**

> 不要回避挫折，但要展示你从中学到了什么。比如"大促前系统差点崩，我临时做了限流保护"，然后重点讲你从中学到了"要提前做容量规划"。

**Q：你的项目和别人有什么不同？**

> 展示你的独特贡献。比如"我主导了全链路追踪改造，引入了 OpenTelemetry"，而不是"我负责订单模块"这种泛泛的描述。

| 复盘维度 | 关键问题 |

| 背景 | 为什么做？解决了什么问题？ |

| 结果 | 数字怎么量化？ |
| 反思 | 学到了什么？下次怎么改进？ |
