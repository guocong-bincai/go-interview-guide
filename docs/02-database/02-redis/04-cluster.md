# Redis 高可用方案：Sentinel 与 Cluster 集群

## 面试官考察意图

考察候选人对 Redis 高可用方案的理解深度。
初级只能说出"主从复制"和"哨兵能自动切换"，高级要能讲清楚 **Sentinel 的主观/客观下线机制、quorum 投票选主流程、Cluster 的哈希槽分片原理、MOVED/ASK 重定向、Gossip 协议、以及 Sentinel 与 Cluster 的选型依据**，并能结合业务场景（数据量、QPS、扩缩容需求）给出具体建议。

---

## 核心答案（30 秒版）

| 方案 | 解决的问题 | 数据量上限 | 扩缩容 | 复杂度 |
|------|------------|------------|--------|--------|
| **主从复制** | 读写分离、数据冗余 | 单机内存上限 | 需人工 | 低 |
| **Sentinel 哨兵** | 自动故障转移、高可用 | 单机内存上限 | 需人工 | 中 |
| **Cluster 集群** | 数据分片 + 高可用 + 水平扩容 | N × 单机内存 | 自动（槽迁移） | 高 |

选型建议：
- **数据量 < 64G、QPS < 10 万** → Sentinel（主从）
- **数据量 > 64G 或 QPS > 10 万** → Cluster

## 深度展开

### 1. Sentinel 哨兵：自动故障转移

**为什么需要哨兵？**

主从架构中，主节点挂了：
- 无法处理写请求
- 无法给从节点同步数据
- 需要人工介入切换，RTO（恢复时间）不可控

哨兵机制解决的问题：**自动完成主从切换**，无需人工干预。

#### 1.1 哨兵的核心职责

哨兵是一个**特殊模式下的 Redis 进程**，主要负责三件事：

```
┌─────────────────────────────────────────────┐
│              哨兵集群（N 个节点）              │
├─────────────────────────────────────────────┤
│  1. 监控（Monitoring）                        │
│     - 每 1s 对所有节点发 PING                 │
│     - 超过 down-after-milliseconds 无响应     │
│       → 标记为「主观下线」                    │
│                                               │
│  2. 选主（Failover）                          │
│     - 多哨兵协作判断「客观下线」               │
│     - Raft 风格的 Leader 选举                 │
│     - 从节点投票选新主                        │
│                                               │
│  3. 通知（Notification）                      │
│     - 告诉客户端新主节点地址                   │
│     - 通过 pub/sub 广播新配置                 │
└─────────────────────────────────────────────┘
```

#### 1.2 主观下线 vs 客观下线

**主观下线（SDOWN）：**
- 单个哨兵对某个节点发 PING，超过 `down-after-milliseconds` 无响应
- 单个哨兵的决定，可能误判（网络抖动）
- **适用所有节点**

**客观下线（ODOWN）：**
- 只针对主节点
- 多个哨兵投票，达到 `quorum` 阈值后认定主节点真正故障
- `quorum` 一般设为 `N/2 + 1`（N 为哨兵数量）
- **只有主节点才需要客观下线**

**为什么主节点需要客观下线，而从节点不需要？**
> 因为从节点挂了只会影响读可用性（还有主节点和其他从节点），不影响数据写入。主节点挂了才真正影响服务。

#### 1.3 故障转移流程

```
步骤 1：主观下线
  哨兵A 对主节点 PING，无响应 → 主观下线
  哨兵A 向其他哨兵发送 is-master-down-by-addr

步骤 2：客观下线 + Leader 选举
  其他哨兵投票 → 达到 quorum（2/3）→ 哨兵A 成为 Leader（候选者）
  Raft 风格：先到先得，拿到多数票即成为 Leader

步骤 3：选新主节点
  从所有健康从节点中，按以下优先级选主：
  1. 优先级最高（replica-priority 配置）
  2. 复制偏移量最大（数据最新）
  3. runid 最小（启动顺序，最早加入集群）

步骤 4：故障转移
  - 向选中的从节点发送 SLAVEOF no one → 变为主节点
  - 向其他从节点发送 SLAVEOF 新主节点 → 重新复制
  - 通知客户端新主节点地址（pub/sub）
  - 继续监视旧主节点 → 恢复后转为从节点
```

**生产经验：**
- 哨兵节点数必须是**奇数**（至少 3 个），避免脑裂（split-brain）
- `quorum` 设为 `N/2 + 1`，确保多数票才能选主
- `quorum` 不是越大越好：如果设为 N，主节点挂了需要全部哨兵同意才能切换，容错性差
- Sentinel 无法自动做**数据分片**，数据量超过单机内存就只能用 Cluster

#### 1.4 脑裂问题

**问题：** 网络分区导致哨兵集群和主节点失联，同时有两个"主节点"对外服务。

```
原主节点（网络分区另一侧，仍认为自己是主）
        ↓
  收到客户端写请求，数据无法同步给从节点
        ↓
  网络恢复后，被哨兵强制降为从节点
        ↓
  这段时间内的写操作丢失！
```

**解决方案：**
```bash
# 1. 配置 min-replicas-to-write + min-replicas-max-lag
# 写操作要求至少 N 个从节点确认复制成功
min-replicas-to-write 1
min-replicas-max-lag 10   # 从节点延迟超过 10s，拒绝写

# 2. 配置 logical-offset（逻辑复制偏移量）监控
# 3. 网络分区时禁止写入（应用层降级）
```

### 2. Redis Cluster：数据分片 + 高可用

**为什么需要 Cluster？**

Sentinel 解决的是"高可用"问题，但无法解决"数据量上限"问题：
- 1 主 + N 从，数据总量 = 1 台机器的内存上限
- 64G 内存的机器，无论挂多少从节点，整个集群只能存 64G 数据

Cluster 的核心思路：**把数据分片（Sharding），存到多台机器上，线性扩展容量**。

#### 2.1 哈希槽（Hash Slot）分片机制

Redis Cluster 把整个数据集划分为 **16384 个槽位**，每个节点负责一部分槽位：

```
客户端请求：SET name "张三"
                ↓
        CRC16("name") % 16384 = 57982
                ↓
槽 57982 由哪个节点管理？→ 集群配置（节点 → 槽位映射）
                ↓
         MOVED 重定向到对应节点
```

**为什么是 16384 个槽？**

1. **詹问题：** 16384 = 2^14，槽位信息用 2 个字节存储（16 bits），传输效率高
2. **足够细：** 16384 个槽均分给 N 个节点，N 最大约 10000，足够覆盖主流集群规模
3. **不是 65536：** 65536 个槽虽然更细，但槽位信息（节点变更时需要广播）会占用过多带宽

**节点扩缩容时，只需迁移槽，不需迁移具体 key：**
```
扩缩容前：节点A 负责槽 0-5460
扩容后：  节点A 负责槽 0-4000，节点D 负责槽 4001-5460
key "user:1" 仍属于槽 57982，无需移动
```

#### 2.2 客户端重定向：MOVED vs ASK

**MOVED 重定向（永久）：**
- 槽的owner节点变了，客户端下次直接去新节点
```go
GET name
// 返回 MOVED 57982 <ip:port>
// 客户端缓存 {槽57982 → 新节点}，下次直接请求新节点
```

**ASK 重定向（临时）：**
- 槽正在迁移中，旧节点无法确定 key 是否已迁走
```go
// 客户端收到 ASK 57982 <ip:port>
// 客户端先去旧节点请求 ASKING，如果 key 已迁移，旧节点返回 ASK 引导客户端去新节点
```
> 注意：ASK 是**指导性重定向**，客户端**仍然在旧节点执行操作**（可能拿到旧值），MOVED 是**强制性重定向**，客户端必须去新节点。

#### 2.3 Gossip 协议：节点间通信

Cluster 中各节点通过 **Gossip 协议** 互相通信，传播：
- 节点存活状态
- 槽位分配信息变更
- 新节点加入 / 节点下线

```
节点A（每 1 秒随机选 3 个节点发 Gossip）
        ↓
        收到 Gossip 的节点继续传播
        ↓
最终所有节点对集群拓扑达成一致
```

Gossip 的特点：**最终一致性**，不保证立即收敛，适合去中心化集群。
- 好处：无需中心节点，单节点故障不影响集群通信
- 坏处：收敛需要几秒到几十秒，有短暂的"不确定状态"

#### 2.4 故障检测与自动转移

```
故障检测：
  节点A 多次联系节点B 无响应 → 标记为「疑似下线」（PFAIL）
        ↓
  通过 Gossip 传播，超过 N/2 个节点都认为B下线 → 标记为「下线」（FAIL）
        ↓
  从节点（持有B负责的槽）收到 FAIL 消息 → 发起故障转移

故障转移：
  从节点执行 SLAVEOF no one → 变为新主节点
        ↓
  更新集群拓扑，通知其他节点
```

**与 Sentinel 的区别：**
| 对比项 | Sentinel | Cluster |
|--------|----------|---------|
| 选主协议 | Raft（中心化协调） | Gossip（去中心化） |
| 故障转移 | Sentinel Leader 执行 | 从节点自发执行 |
| 数据分片 | 无 | 有（哈希槽） |
| 客户端路由 | 需要 Sentinel 客户端 | 支持 smart 客户端直连 |

### 3. Sentinel vs Cluster：选型决策树

```
你的业务数据量是多少？
        │
        ├── < 64G
        │     ├── QPS < 10 万
        │     │     └── 推荐：Sentinel（1 主 + 2 从）
        │     │           优点：配置简单，够用
        │     │           缺点：无法水平扩容
        │     │
        │     └── QPS > 10 万
        │           └── 推荐：Cluster（3 主 + 每主 1 从）
        │                 优点：可水平扩容
        │                 缺点：运维复杂度高，跨槽操作需注意
        │
        └── > 64G
              └── 推荐：Cluster（根据数据量决定主节点数）
                        经验公式：主节点数 = ceil(数据量GB / 64)
                        每主至少 1 个从节点保证高可用
```

**生产经验：**
- Cluster 至少 **6 个节点**（3 主 + 3 从），主节点数建议奇数
- 避免"热点槽"：槽分配尽量均匀，定期监控 `redis-cli --cluster check` 输出
- 扩缩容使用 `redis-cli --cluster reshard` 在线迁移，不影响服务
- 客户端必须使用 **smart 客户端**（如 Go 的 go-redis/redis.v9），不支持 dumb 客户端

### 4. 面试高频追问

**Q1：Sentinel 和 Cluster 能不能一起用？**
> 可以，但没必要。Cluster 本身已包含主从复制 + 故障转移，Sentinel 只适合非 Cluster 架构。

**Q2：Cluster 某主节点挂了，从节点会自动升级吗？**
> 会。从节点发现主节点 FAIL 后，会自动发起故障转移，升级为新主节点，其他从节点自动指向新主。

**Q3：扩缩容时业务会有影响吗？**
> 在线槽迁移期间，通过 ASK 重定向保证请求不丢，但可能有短暂延迟。建议在业务低峰期操作。

**Q4：Cluster 的 ASK 重定向和 MOVED 重定向有什么区别？**
> MOVED 是槽已完全迁移，客户端应直接更新路由缓存；ASK 是槽正在迁移中，客户端仍向旧节点发请求（旧节点引导客户端去新节点尝试）。

**Q5：Sentinel 的 quorum 配置有什么坑？**
> quorum 太小（1）容易误判，quorum 太大（等于 N）则任何节点挂了都无法切换。建议 `quorum = N/2 + 1`。
