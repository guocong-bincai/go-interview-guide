# 面试高频场景题（开放性问题）

> 考察频率：★★★★★  优先级：P0

> 这类题没有标准答案，考察的是系统性思维、表达能力和真实项目经验。每道题给出「答题框架」和「加分点」。面试官关注的是你思考问题的方式，而不是背一个标准答案。

---

## 1. 如何设计一个高可用的系统

### 面试官考察意图

考察候选人对「可用性」这件事的系统性理解。不是问你用了什么工具，而是问你**面对故障时的思路**。高级工程师的答案是：故障前预防 → 故障中止损 → 故障后恢复，形成闭环。

### 答题框架

**核心思路：从「防止故障」到「容忍故障」到「快速恢复」**

```
高可用设计框架
════════════════════════════════════════════════════

故障前预防（Prevent）
    ├── 冗余：无单点（多副本、集群部署）
    ├── 隔离：故障域分离（不同机房/机架/进程）
    ├── 限流降级：超出能力自动保护
    └── 监控告警：问题发生前就发现

故障中容忍（Resist）
    ├── 熔断器：防止故障蔓延
    ├── 重试+幂等：瞬时故障自动恢复
    └── 降级：核心功能优先，非核心牺牲

故障后恢复（Recover）
    ├── 快速发现：监控+告警
    ├── 快速止血：切流/回滚/降级
    ├── 快速恢复：自动化 failover
    └── 复盘改进：根因分析+改进项

════════════════════════════════════════════════════
```

### 量化指标

| 可用性 | 每年停机时间 | 典型场景 |
|--------|-------------|---------|
| 99% | 3.65天 | 小公司内部系统 |
| 99.9%（3个9）| 8.76小时 | 创业公司核心服务 |
| 99.99%（4个9）| 52分钟 | 中大型公司核心服务 |
| 99.999%（5个9）| 5分钟 | 金融/电信级系统 |

### 加分点

1. **混沌工程**：主动注入故障，验证系统韧性（Netflix Chaos Monkey、ChAP）
2. **同城双活**：两个机房同时对外服务，任一机房故障秒级切换
3. **接口限流要区分来源**：防止单个用户的流量压垮整体服务

### 参考话术

> 「高可用的核心不是不出故障，而是让故障发生时用户无感知。我之前负责的支付系统做了几件事：1）前置限流模块，超阈值直接返回503，不让流量打垮后端；2）核心链路全部集群部署，任一节点挂了自动摘流；3）第三方支付渠道做了熔断降级，支付通道不可用时切到备用渠道。整体SLA做到了99.99%。」

## 2. 你们系统最大的技术债是什么，怎么还的

### 面试官考察意图

考察候选人有没有**自我反思能力**和**推动改进的经历**。技术债是所有团队都有的，面试官想看你：1）能不能识别出技术债；2）有没有能力推动解决；3）解决过程中如何平衡业务压力和技术价值。

### 答题框架

**识别 → 评估 → 方案 → 落地 → 结果**

```
技术债治理闭环
═══════════════════════════════════════════════════

识别（Symptom）
    └── 这个债的症状是什么？（慢？难扩展？线上故障？）
        「接口P99延迟从50ms涨到了500ms」

评估（Impact）
    └── 影响的业务范围、频率、代价
        「每日峰值期有5%用户受影响，客诉增加」

方案（Solution）
    └── 渐进式改良 vs 重写
        「历史数据量大，不适合重写，选择局部优化」

落地（Milestone）
    └── 拆成小里程碑，逐步灰度
        「第一期优化DB索引，第二期优化缓存结构」

结果（Result）
    └── 量化改善指标
        「P99从500ms降到80ms，客诉归零」

═══════════════════════════════════════════════════
```

### 加分点

1. **说明为什么当时会产生这个债**：让面试官理解不是你的锅，而是历史局限性
2. **你主动发现了这个问题**：说明你有技术owner意识，不只是执行者
3. **推动了团队一起解决**：说明你有跨团队协作能力

### 参考话术

> 「最大的技术债是我们订单服务的数据库没有分库分表，单表数据到了5000万条。每次大促前我们都要提前一个月做主从延迟监控。解决方案是分库分表，用 ShardingSphere，按 user_id 路由。迁移过程中最难的是历史数据的双写验证，我们做了3轮影子流量对比才确认数据一致性没问题。最终单表数据降到500万，大促期间数据库完全没压力。」

## 3. 如果让你重新设计现在的系统，你会怎么做

### 面试官考察意图

考察候选人的**系统思维高度**和**架构能力**。这个问题没有标准答案，关键是看你：1）能不能识别现有系统的问题；2）重新设计的思路是否合理；3）能不能意识到重写的风险和成本。

### 答题框架

**问题 → 设计原则 → 核心改动 → 风险与权衡**

```
重新设计思路
═══════════════════════════════════════════════════

先说问题（不要全盘否定现有系统）
    └── 核心问题是什么？（扩展性？性能？维护成本？）
        「DB单表太大，水平扩展困难」

再说设计原则
    └── 无状态化 / 读写分离 / 事件驱动 / 微服务拆分
        「核心链路无状态，状态推到分布式存储」

然后说改动最大的地方
    └── 不要说全部重写，说关键改动
        「分库分表 + 引入消息队列解耦」

最后说风险
    └── 重写风险、迁移成本、过渡方案
        「迁移期间双写，新系统验证后逐步切流量」

═══════════════════════════════════════════════════
```

### 加分点

1. **不要全盘否定现有系统**：说明当时的决策背景，重写代价和风险
2. **提到演进式重构**： strangler fig pattern（绞杀者模式），逐步替换
3. **量化重写的收益和成本**：让面试官看到你做过可行性评估

### 参考话术

> 「我不会重写，而是局部改造。核心问题是订单服务耦合了库存和促销逻辑。重新设计的话，我会：1）先把库存和促销拆成独立服务，用消息队列做事件驱动解耦；2）订单服务只保留核心流程，其他逻辑通过订阅事件异步处理；3）数据库按业务拆分，订单、库存、促销各一个库。好处是不需要大停机，可以按模块逐步迁移，风险可控。」

## 4. 线上出了 P0 故障，你怎么处理

### 面试官考察意图

考察候选人的**应急响应能力**和**故障处理经验**。P0 是最高级别故障，面试官想看你：1）能不能在压力下冷静处理；2）有没有标准化的事故处理流程；3）能不能做到止血+定位+恢复的优先级排序。

### 答题框架（CLAER 模型）

```
P0 故障处理：CLAER
═══════════════════════════════════════════════════════════

C - Cut（止血，第一优先）
    └── 目标是让用户尽快恢复，不是找根因
        「切流 / 回滚代码 / 降级非核心功能」
        「如果DB挂了，切到只读副本；如果缓存挂了，关闭缓存直接读DB」

L - Locate（定位）
    └── 止血后开始找根因
        「看监控曲线拐点 → 查变更记录 → 链路追踪定位具体服务」

A - Action（修复）
    └── 最小改动原则，不要引入新问题
        「修复后一定要验证，不要以为修好了就不管了」

E - Evacuate（恢复）
    └── 逐步放量，观察指标
        「先1%流量，观察5分钟，没问题再10%...」

R - Review（复盘）
    └── 48小时内完成复盘报告
        「根因、改进项、后续SLO承诺」

═══════════════════════════════════════════════════════════
```

### 真实案例话术

> 「我经历过一次Redis Cluster脑裂导致的P0。凌晨3点收到告警，接口P99从100ms涨到10s。
> **止血**：先切到本地缓存降级，保证服务不挂。
> **定位**：查监控发现有两个master同时存在，脑裂了。根因是网络抖动导致3个节点同时认为其他节点死了。
> **修复**：手动kill掉一个master节点，让正常节点重新选主。
> **恢复**：观察10分钟，确认数据一致后关闭降级。
> **复盘**：后来加了Redis集群脑裂检测告警，脑裂后自动发钉钉告警，不再等监控发现。」

### 加分点

1. **有具体的量化数据**：恢复时间、根因、影响范围
2. **有后续改进措施**：不是处理完就完了，要说改了什么防止再发生
3. **知道什么时候升级**：什么时候该拉更多人进来，什么时候该通知领导

## 5. 你负责的系统 QPS 从 1000 提升到 10 万，怎么做

### 面试官考察意图

考察候选人的**性能优化全链路思维**。不是问某个点怎么优化，而是整个从小到大的扩展路径。这道题考察的是你有没有系统性的性能优化方法论。

### 答题框架

**找瓶颈 → 横向扩展 → 纵向优化 → 架构升级**

```
QPS 1000 → 10万 扩容路径
═══════════════════════════════════════════════════════════

阶段1: 1000 → 5000 QPS
├── 瓶颈：单个DB连接池瓶颈
└── 手段：连接池调优（maxOpen/maxIdle）+ 加索引

阶段2: 5000 → 2万 QPS
├── 瓶颈：DB CPU/IO 达上限
└── 手段：读写分离（从库分担读流量）

阶段3: 2万 → 5万 QPS
├── 瓶颈：单机带宽/内存达上限
└── 手段：水平扩展（加机器）+ Redis缓存热点数据

阶段4: 5万 → 10万 QPS
├── 瓶颈：DB写入成为瓶颈
└── 手段：分库分表（按业务拆分写入）+ 异步化（非核心链路MQ处理）

辅助手段（每个阶段都要做）
├── 前端限流（防止突发流量压垮系统）
├── 服务熔断（防止下游故障蔓延）
└── 监控告警（每个瓶颈的发现都依赖监控）

═══════════════════════════════════════════════════════════
```

### 各层优化要点

| 层级 | 常见瓶颈 | 优化手段 |
|------|---------|---------|
| 入口层 | 带宽打满 | CDN、请求压缩 |
| 网关层 | 连接数上限 | 多实例 + L4负载均衡 |
| 应用层 | CPU、内存 | 水平扩容、无状态化 |
| 缓存层 | Redis QPS上限 | 分片集群、热点分离 |
| DB层 | 写入瓶颈 | 分库分表、异步写入 |
| 整体 | 跨服务调用 | 批量接口、异步消息 |

### 加分点

1. **每步都有量化指标**：不是说加机器就完了，要说加到性能提升多少
2. **知道什么时候该做什么**：不是上来就分库分表，成本很高
3. **有回滚方案**：每个措施都要能退回去

### 参考话术

> 「我不会上来就改架构，先找瓶颈。
> 第一步：Profiling。接口P99高，我用pprof分析CPU和内存，发现JSON序列化占用30%CPU，换了sonic后降低到15%。
> 第二步：加缓存。热点数据放Redis，命中率80%，DB QPS从2万降到4000。
> 第三步：读写分离。报表类查询打到从库，主库QPS再降一半。
> 第四步：水平扩容。因为应用层是无状态的，直接加机器，QPS翻倍。
> 最终从1000到了8万，还差2万才到目标。瓶颈是支付写库，走了分库分表方案。」
