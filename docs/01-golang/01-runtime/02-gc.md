# GC 垃圾回收机制

## 面试官考察意图

考察候选人对 Go 内存管理的深度理解。
初级只答"标记清除"，高级要能讲清楚**三色标记的并发安全问题、写屏障如何解决、STW 在哪些阶段、Go 各版本的优化历程**，以及如何在生产中调优 GC。

---

## 核心答案（30 秒版）

Go 使用**三色并发标记清除（Tri-color Concurrent Mark and Sweep）**算法，辅以**混合写屏障（Hybrid Write Barrier）**保证并发安全。

核心特点：
- **并发标记**：GC 与用户代码同时运行，大幅减少 STW 时间
- **非移动式**：不压缩内存，无需更新指针（代价是内存碎片）
- **分代假设不成立，改用分 span**：通过 `mspan` 管理不同大小对象

当前（Go 1.18+）STW 时间通常在 **< 1ms**。

## 深度展开

### 1. 三色标记算法

对象被标记为三种颜色：

| 颜色 | 含义 |
|------|------|
| **白色** | 未被扫描，GC 结束后仍为白色 = 垃圾，会被回收 |
| **灰色** | 已被发现（加入待扫描队列），但其引用的对象未被扫描 |
| **黑色** | 已扫描完毕，其所有引用对象也已处理 |

**标记过程：**

```
初始：所有对象白色

1. STW Start（极短）
   ├── 标记根对象（栈变量、全局变量、寄存器）为灰色
   └── 开启写屏障

2. 并发标记（与用户代码并行）
   ├── 取灰色对象 → 扫描其引用 → 将引用对象变灰 → 自身变黑
   └── 重复直到灰色队列为空

3. STW Mark Termination（极短）
   ├── 重新扫描在并发阶段被修改的对象（写屏障记录）
   └── 关闭写屏障

4. 并发清扫（与用户代码并行）
   └── 回收所有白色对象的内存（归还 mspan）
```

### 2. 并发标记的安全问题

并发标记期间，用户代码可能修改对象引用，导致**对象漏标**：

```
初始状态：A(黑) → B(灰) → C(白)

用户代码执行：
  A.ref = C   // A 直接引用 C
  B.ref = nil // B 断开对 C 的引用

结果：A 已经是黑色不会重扫，C 永远是白色
     → C 被错误回收！（悬空指针，程序崩溃）
```

漏标的充要条件（两者同时满足才会漏标）：
1. 黑色对象新增了对白色对象的引用
2. 该白色对象与灰色对象的引用路径被切断

### 3. 写屏障（Write Barrier）

写屏障是**赋值操作的钩子**，在指针写入时记录变化，防止漏标。

**Go 1.7 之前：Dijkstra 插入写屏障**

```
写操作 *slot = ptr 时：
    shade(ptr)  // 将新引用对象标灰
```

问题：**栈上的写操作不加写屏障**（性能原因），导致 Mark Termination 需要 STW 重扫所有栈。

**Go 1.8+：混合写屏障（Hybrid Write Barrier）**

```
写操作 *slot = ptr 时：
    shade(*slot)  // 将旧值标灰（删除屏障）
    shade(ptr)    // 将新值标灰（插入屏障）
```

**核心优势：栈上对象在 GC 开始时全部标黑，无需重扫栈，STW 时间从百ms级降到 < 1ms。**

```go
// 伪代码：writeBarrier 的实现
func writePointer(slot *unsafe.Pointer, ptr unsafe.Pointer) {
    if writeBarrier.enabled {
        shade(*slot) // 旧值
        shade(ptr)   // 新值
    }
    *slot = ptr
}
```

### 4. GC 触发时机

```go
// 三种触发方式
1. 堆内存增长到阈值
   // 上次 GC 后存活对象大小 * (1 + GOGC/100)
   // GOGC=100（默认）意味着堆翻倍时触发

2. 距上次 GC 超过 2 分钟（sysmon 强制触发）

3. 手动触发
   runtime.GC()
```

```go
// 调整 GC 触发频率
import "runtime/debug"

// GOGC=200：堆增长 200% 才触发，减少 GC 频率，换取更高内存占用
debug.SetGCPercent(200)

// Go 1.19+：设置内存上限，超过时强制 GC
debug.SetMemoryLimit(4 * 1024 * 1024 * 1024) // 4GB
```

### 5. Go GC 演进历史

| 版本 | 关键改进 | STW 时间 |
|------|----------|----------|
| Go 1.1 | 串行 STW GC | 数百 ms ~ 数秒 |
| Go 1.5 | 并发三色标记 | 数十 ms |
| Go 1.6 | 减少重扫范围 | 数 ms |
| Go 1.8 | **混合写屏障**，消除栈重扫 STW | < 1ms |
| Go 1.14 | 异步抢占，GC 更及时 | < 1ms |
| Go 1.18 | 软内存限制（`SetMemoryLimit`） | < 1ms |
| Go 1.21 | GC pacing 优化，更平稳 | < 1ms |
| Go 1.25 | **Green Tea GC（实验）**，改善小对象标记和扫描的局部性，GC 开销降低 10~40% | < 1ms |
| Go 1.26 | **Green Tea GC 默认启用**，amd64 平台利用 SIMD 加速小对象扫描，GC 开销再降约 10% | < 1ms |

### 6. 内存分配：tcmalloc 思想

Go 内存分配器基于 tcmalloc，核心层次：

```
用户程序
    │ 小对象 (≤32KB)          大对象 (>32KB)
    ▼                              ▼
  mcache                        mheap
（P 本地缓存，无锁）          （全局堆，有锁）
    │
    ▼ 不足时向上申请
  mcentral
（按 span 大小分类，有锁）
    │
    ▼ 不足时向上申请
  mheap → OS（mmap）
```

对象大小分为 **67 个 size class**（8B ~ 32KB），减少内存碎片：

```
8B, 16B, 24B, 32B, 48B, 64B, 80B, 96B, 112B, 128B, ...
```

### 7. 生产经验：GC 调优

**场景 1：服务 P99 延迟毛刺，怀疑 GC**

```bash
# 开启 GC trace
GODEBUG=gctrace=1 ./myserver

# 输出示例：
# gc 1 @0.012s 7%: 0.018+1.3+0.076 ms clock
#         ↑        ↑    ↑   ↑   ↑
#     GC编号  占用CPU  STW 并发 STW
```

```go
// 或用代码采集 GC stats
var stats runtime.MemStats
runtime.ReadMemStats(&stats)
fmt.Printf("GC次数: %d, 上次STW: %v\n",
    stats.NumGC, time.Duration(stats.PauseNs[(stats.NumGC+255)%256]))
```

**场景 2：内存分配过频繁（GC 压力大）**

```go
// 优化前：每次请求分配新 buffer
func handle(r *http.Request) {
    buf := make([]byte, 4096)
    // ...
}

// 优化后：sync.Pool 复用
var bufPool = sync.Pool{
    New: func() interface{} { return make([]byte, 4096) },
}
func handle(r *http.Request) {
    buf := bufPool.Get().([]byte)
    defer bufPool.Put(buf)
    // ...
}
```

**场景 3：容器环境 OOM**

```go
// Go 1.19+ 设置内存上限，防止 OOM 被 kill
// 建议设为容器内存限制的 90%
import "runtime/debug"

func init() {
    // 容器 memory limit 为 2GB，设置 1.8GB 触发 GC
    debug.SetMemoryLimit(1800 * 1024 * 1024)
}
```

## 高频追问

**Q：为什么 Go GC 不做内存压缩（移动对象）？**

移动对象需要更新所有指向该对象的指针，而 Go 支持 `unsafe.Pointer` 和 CGO，运行时无法追踪所有指针，因此不做移动。代价是存在内存碎片，但通过 `mspan` 的 size class 机制已大幅缓解。

**Q：GOGC 调大或调小分别适用什么场景？**

| GOGC 值 | 效果 | 适用场景 |
|---------|------|----------|
| 调小（如 50） | 更频繁 GC，内存占用低，CPU 开销高 | 内存敏感，如嵌入式 |
| 默认（100） | 堆翻倍触发，均衡 | 大多数服务 |
| 调大（如 400） | 更少 GC，内存占用高，延迟低 | 延迟敏感，如游戏服务器 |
| off（-1） | 关闭自动 GC | 仅手动 `runtime.GC()` |

**Q：STW 期间程序完全停止吗？**

是的，STW 期间所有 goroutine 停止运行。但现代 Go 的 STW 只发生在：
1. GC 开始（设置写屏障）：< 100μs
2. Mark Termination（混合写屏障消除了栈重扫）：< 500μs

**Q：sync.Pool 和 GC 的关系？**

`sync.Pool` 中的对象**在每次 GC 时都会被清空**。这是故意的设计，防止 Pool 无限增长导致内存泄漏。如果需要跨 GC 保留对象，应使用带 TTL 的自定义缓存。

**Q：如何判断某个对象分配在堆还是栈？**

```bash
go build -gcflags='-m -m' ./... 2>&1 | grep "escapes to heap"
```

**Q：Go 1.26 新增的 goroutine leak profile 是什么？**

Go 1.26 在 `runtime/pprof` 中新增了实验性 `goroutineleak` profile 类型，可通过 `GOEXPERIMENT=goroutineleakprofile` 启用，配合 `net/http/pprof` 的 `/debug/pprof/goroutineleak` 端点使用。

其原理是：GC 追踪每个 goroutine 阻塞在哪个同步原语上（channel、Mutex、Cond 等），如果该同步原语本身已从所有可运行的 goroutine 不可达，则该 goroutine 永远无法被唤醒，判定为泄漏。

典型泄漏案例：
```go
ch := make(chan result)
for _, w := range ws {
    go func() { ch <- processWorkItem(w) }()
}
for range len(ws) {
    r := <-ch
    if r.err != nil { return nil, r.err } // ⚠️ 提前返回时，未读完的 goroutines 泄漏
}
```

该 profile 零开销（仅在主动采集时才触发检测），计划在 Go 1.27 默认启用。

## Go 1.24 新增：`runtime.AddCleanup` — 更安全的对象清理机制

> Go 1.24 引入了 `runtime.AddCleanup`，这是对经典 `runtime.SetFinalizer` 的全面升级，面试中正在变得越来越高频。

### SetFinalizer 的历史问题

`runtime.SetFinalizer` 是 Go 历史上用于对象清理的机制，但有三个典型坑点：

| 问题 | 说明 |
|------|------|
| **循环引用导致泄漏** | 对象 A 引用 B，B 通过 finalizer 引用 A，GC 无法回收，形成孤岛 |
| **只能绑定一个清理函数** | 每个对象只能有一个 finalizer，无法附加多个 |
| **延迟执行时机不明确** | finalizer 执行时机不确定，可能导致资源泄漏 |

### `AddCleanup` 的改进

```go
import "runtime"

// ✅ Go 1.24+：用 AddCleanup
type Cache struct {
    data []byte
    file *os.File
}

func NewCache(path string) *Cache {
    f, _ := os.Open(path)
    c := &Cache{
        data: make([]byte, 1024),
        file: f,
    }
    // AddCleanup：对象不可达时自动调用清理函数
    runtime.AddCleanup(c, func() {
        // 关闭文件
        if c.file != nil {
            c.file.Close()
        }
        // 释放内存（切片自动 GC）
        c.data = nil
    })
    return c
}
```

**`AddCleanup` vs `SetFinalizer` 对比：**

| 维度 | `SetFinalizer` | `AddCleanup` |
|------|----------------|---------------|
| 清理函数数量 | 只能 1 个 | 可以多个 |
| 循环引用 | 可能导致泄漏（对象图形成环）| **不导致泄漏**（GC 能处理）|
| 内部指针 | 只能指向对象自身 | 可以指向对象的内部指针 |
| 延迟释放 | 可能延迟对象释放 | **不延迟对象释放** |
| 执行时机 | 不确定 | 对象不可达后尽快执行 |

### 典型面试题

**Q：`SetFinalizer` 的循环引用问题是怎么产生的？**

```go
// ⚠️ 循环引用导致 finalizer 泄漏
type A struct { b *B }
type B struct { a *A }

objA := &A{b: &B{a: objA}} // A 和 B 互相引用，且 finalizer 持有对方引用
runtime.SetFinalizer(objA, func(a *A) { ... })

// objA 和 objB 形成孤岛：虽然外部已无引用，
// 但 finalizer 持有内部引用，GC 无法回收
```

**Q：生产环境中何时该用 `AddCleanup` 而不是 `defer`？**

- **`defer`**：适用于函数内的确定性清理（函数返回时执行）
- **`AddCleanup`**：适用于**对象生命周期超过创建它的函数**的场景，如：
  - 缓存中的大对象（缓存驱逐时清理）
  - CGO 分配的非 Go 内存
  - 文件句柄、数据库连接等需要延迟清理的资源
